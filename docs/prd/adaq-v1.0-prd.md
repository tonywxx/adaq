## 1. 文档目的

本文档是 ADAQ V1.0 的产品与软件需求基线，供产品设计、架构设计、AI 编码、测试和发布验收共同使用。所有实现任务必须追溯到本文档中的目标、需求编号或验收标准。

本文档中的“必须”表示 V1.0 发布阻断项；“应当”表示高优先级要求，只有在记录原因和降级影响后才能延期；“不得”表示明确禁止的行为。

## 2. Problem Statement

专业及进阶个人量化用户需要在多个加密货币中心化交易所之间获取一致、可验证的市场数据，进行可复现的因子研究和策略回测，并在不承担真实资金风险的环境中验证订单、费用、滑点和交易所规则。

现有工具通常存在以下问题：

- 数据来源、缺口、规则版本和回测假设不透明，结果难以复现。
- 向量化研究与事件驱动交易模拟相互割裂，研究结果无法可靠过渡到订单级验证。
- 不同交易所的交易对、精度、费率和订单语义被过度简化。
- 第三方策略往往拥有过大的本地权限，或必须交付源码。
- 本地模型和商业模型的接入方式不统一，API 密钥容易暴露在前端或日志中。
- 研究平台过早混入真实自动交易、付费商城和复杂衍生品，扩大安全、监管和工程风险。

## 3. Solution

ADAQ V1.0 将提供一个本地优先、桌面端运行的加密货币现货量化研究平台，完整覆盖：

1. OKX 与 Bitget 现货市场数据采集、标准化和质量管理。
2. Python/DuckDB 向量化因子研究。
3. Rust 事件驱动策略回测和确定性本地模拟交易。
4. OKX 与 Bitget 官方 Demo Trading 环境验证。
5. 基于 WebAssembly Component Model 的因子和策略沙箱。
6. 支持 OpenAI-compatible、Anthropic-compatible 和本地 LLM 的只读 AI 研究助手。

V1.0 不执行真实资金交易，不提供付费 Trader Store，不支持期货、合约、期权、杠杆或做空。

## 4. 产品目标

### 4.1 核心目标

- 建立可信、可复现、可审计的本地量化研究和回测基础设施。
- 用 OKX 完成首个端到端纵向切片，再以 Bitget 验证连接器抽象。
- 让研究代码、策略代码、数据集、费用规则和模拟假设均可版本化。
- 让同一套 Rust OMS、账本、组合和风控逻辑服务于历史回测、实时本地模拟和交易所 Demo。
- 建立可供未来 Trader Store 使用的版本化 WIT 接口、签名包和能力沙箱。
- 在不暴露用户密钥和本地数据的前提下提供可替换的 AI 助手。

### 4.2 成功指标

- OKX 和 Bitget 的公共行情、交易规则及 Demo API 均通过连接器契约测试。
- 全市场最近 12 个月的 1 分钟 K 线可渐进同步，并可查询覆盖率和缺口。
- 对同一数据集、策略、参数、初始资金和随机种子，重复回测产生相同结果哈希。
- 本地模拟和 Demo Trading 在界面、数据模型和日志中可明确区分。
- 所有第三方 WASM 组件均无法绕过宿主 OMS 或访问未授权系统资源。
- 应用、Worker 或 WebSocket 异常后，持久化任务和账户状态能够恢复或明确进入失败状态。
- 用户能够使用 OpenAI-compatible、Anthropic-compatible 或 localhost 模型完成流式只读问答。
- macOS 和 Windows 均通过全新安装、升级、迁移和卸载验证。

## 5. 用户与角色

### 5.1 Quant Researcher

使用 Python、SQL 和 Notebook 研究因子、筛选参数、比较实验，并要求数据来源与结果可复现。

### 5.2 Strategy Developer

使用 Rust SDK 开发因子或策略，将其编译为 WASM Component，并在受限沙箱中进行回归和性能验证。

### 5.3 Simulated Trader

使用本地模拟或交易所 Demo 环境验证订单意图、费用、滑点、部分成交、资产变化和风险限制。

### 5.4 Platform Operator

维护本地数据、任务、连接器、设备会话、SDK 版本和安全策略；V1.0 中该角色通常也是本机用户。

## 7. 总体架构

V1.0 采用本地优先的模块化单体，不采用微服务、Redis 或 RabbitMQ。

### 7.1 运行单元

- React WebView：只负责展示、输入和交互状态。
- Tauri Rust Host：负责认证编排、权限、网络、数据目录、应用服务和安全存储。
- Rust Backtest/Strategy Worker：负责 CPU 密集型回测、模拟和 WASM 执行。
- Python Research Worker/SDK：负责用户主动启动的研究任务，不进入策略运行路径。
- Supabase Control Plane：只负责认证、用户资料、设备会话和设备撤销。

### 7.2 安全边界

- React 不得直接调用 CEX 或 LLM，不得持有 CEX、LLM 或 Supabase Refresh Token。
- Rust Host 是所有外部网络和系统密钥库访问的唯一可信入口。
- Python 研究进程只能获得用户授权的数据路径，不得获得系统密钥。
- 第三方组件只能通过版本化 WIT 接口与宿主交互。
- 本机不得启动对局域网或公网开放的内部 HTTP 服务。
- 长任务必须运行在可取消、可监督的 Worker 中，不能阻塞 WebView 或 Tauri 主线程。

### 7.3 深模块边界

- Identity and Device Service：封装登录、令牌、设备和离线宽限。
- Market Data Service：封装下载、实时流、标准化、质量与目录。
- Exchange Adapter Port：统一交易所能力，同时保留原生字段。
- Research Service：封装数据访问、实验 Manifest 和因子结果导入。
- Backtest Engine：封装确定性事件推进、OMS、成交和账本。
- Simulation Service：封装本地模拟与交易所 Demo 环境。
- Component Runtime：封装 WIT、Wasmtime、能力和资源限制。
- AI Gateway：封装协议差异、上下文授权、脱敏和流式响应。
- Task Service：封装任务状态机、检查点、恢复和进度事件。

## 8. 功能需求

### 8.1 身份与设备

- **AUTH-001** 系统必须在首次使用时要求 Supabase 登录。
- **AUTH-002** 系统必须支持邮箱 Magic Link/OTP、Google OAuth 和 GitHub OAuth。
- **AUTH-003** OAuth 必须使用系统浏览器、PKCE 和应用 Deep Link 回调。
- **AUTH-004** Refresh Token 必须存入系统密钥库，不能写入 SQLite 或前端存储。
- **AUTH-005** 系统必须注册设备公钥、设备名称、平台和最近验证时间。
- **AUTH-006** 已验证设备必须获得 7 天离线宽限，用于访问已有本地数据和回测。
- **AUTH-007** 离线宽限结束后，系统必须要求联网验证，不能无限期续期。
- **AUTH-008** 设备撤销必须在下一次联网时清除本机会话及受保护内容密钥。
- **AUTH-009** 所有 Supabase 业务表必须启用 RLS，桌面包不得包含 Service Role Key。

### 8.2 交易所与产品主数据

- **CEX-001** V1.0 必须支持 OKX Spot 和 Bitget Spot。
- **CEX-002** 每个交易所必须分别实现公共数据、历史数据、实时数据和 Demo 私有 API 能力。
- **CEX-003** 产品标识必须包含 Venue、MarketType 和 NativeSymbol。
- **CEX-004** 系统不得仅按 Base/Quote 合并不同交易所的产品和流动性。
- **CEX-005** 系统必须保存交易状态、价格精度、数量精度、最小数量、最小名义金额和费率信息。
- **CEX-006** 交易所原生响应中的必要字段必须保留，以支持故障诊断和未来协议演进。
- **CEX-007** 每个连接器必须实现独立限流、时钟校准、重试、心跳和断线重连。
- **CEX-008** 连接器必须区分可重试错误、永久错误、认证错误和限流错误。
- **CEX-009** 生产下单端点必须在 V1.0 构建和运行配置中被禁止。
- **CEX-010** 测试必须证明 Demo 请求无法被路由到生产下单端点。

### 8.3 历史数据与实时行情

- **DATA-001** 首次初始化必须同步 OKX 和 Bitget 的全部活跃现货交易对元数据。
- **DATA-002** 系统必须支持渐进下载全市场最近 12 个月的 1 分钟 K 线。
- **DATA-003** 用户关注的交易对必须优先同步，其他交易对在后台补齐。
- **DATA-004** 用户必须能够为单个交易对请求交易所可提供的更早历史数据。
- **DATA-005** 用户必须能够为选定交易对启用逐笔成交采集。
- **DATA-006** 逐笔成交默认保留 30 天，用户可以修改保留期。
- **DATA-007** 无法通过官方 API 获取的历史逐笔数据必须标记为不可用，不得推算或伪造。
- **DATA-008** 系统必须通过 REST 快照和 WebSocket 增量维护实时 L2 订单簿。
- **DATA-009** 检测到序列缺口、乱序或重连时，系统必须废弃旧订单簿并重新同步。
- **DATA-010** V1.0 不得把 L2 数据作为可历史回放数据持久化。
- **DATA-011** 标准市场事件必须包含 exchange_time、receive_time、ingest_time、sequence、source、connector_version 和 quality_flags。
- **DATA-012** 价格、数量、金额和费用必须使用 Decimal 或定点类型，不能使用二进制浮点记账。
- **DATA-013** K 线和逐笔成交必须按 Venue、Market、Symbol、数据类型和日期分区存储为 Parquet。
- **DATA-014** SQLite 必须保存覆盖范围、下载任务、Schema 版本、校验和、缺口和磁盘占用。
- **DATA-015** DuckDB 必须能够直接查询 Parquet，不要求将全部历史数据复制进数据库。
- **DATA-016** 数据质量报告必须识别缺失、重复、乱序、异常时间戳和交易规则变化。
- **DATA-017** 删除或缩短保留期前，系统必须展示受影响的研究和回测记录。
- **DATA-018** 历史数据访问必须经 HistoricalDataProvider 接口，以便后续接入中心化数据湖。

### 8.4 Python 研究与因子分析

- **RES-001** 系统必须提供版本化 Python SDK、CLI 和项目模板。
- **RES-002** Python SDK 必须支持只读访问用户授权的 DuckDB/Parquet 数据。
- **RES-003** 用户可以在 JupyterLab、VS Code 或 PyCharm 中使用 SDK，V1.0 不内置 IDE。
- **RES-004** 每次实验必须输出包含数据哈希、参数、SDK 版本、代码哈希和时间范围的 Research Manifest。
- **RES-005** 桌面端必须验证 Manifest 和关联文件后再导入研究结果。
- **RES-006** 系统必须支持 IC、Rank IC、分组收益、换手率、稳定性和因子相关性分析。
- **RES-007** Python 研究可以输出因子序列、目标权重、信号和报告。
- **RES-008** 外部 Python 进程不得获取 OS Keychain 或调用内部 Tauri 命令。
- **RES-009** Python 研究结果不能被标记为可运行沙箱策略，除非另行实现并编译 Rust/WASM 策略。

### 8.5 WASM 因子与策略

- **WASM-001** 系统必须定义并版本化 `adaq:factor@1` 和 `adaq:strategy@1` WIT 世界。
- **WASM-002** Factor 组件必须支持初始化、批量计算和标准因子序列输出。
- **WASM-003** Strategy 组件必须支持初始化、事件处理、状态快照、状态恢复和关闭。
- **WASM-004** Strategy 只能输出 OrderIntent，不得直接创建交易所订单。
- **WASM-005** 组件默认不得访问网络、文件系统、环境变量、进程和真实系统时钟。
- **WASM-006** 宿主只能提供规范化行情、因子值、只读组合状态、逻辑时钟、确定性随机数、结构化日志和状态存取。
- **WASM-007** 每个组件必须配置内存、Fuel、Epoch 超时、调用期限、日志、状态和输出限制。
- **WASM-008** 连续超时或失败的组件必须被熔断并生成可诊断错误。
- **WASM-009** 组件执行失败不得导致桌面主进程退出。
- **WASM-010** 策略包必须包含 Manifest、加密 Component、Schema、签名和使用说明。
- **WASM-011** 策略包必须使用 Ed25519 验证完整性和作者身份。
- **WASM-012** V1.0 必须支持私有签名包，并以 Beta 形式支持设备绑定加密。
- **WASM-013** 内容解密只能发生在受监督 Worker 内存中，不得写出明文组件文件。
- **WASM-014** 平台不得要求策略作者上传源码；只处理编译后的 Component 和元数据。
- **WASM-015** 产品文案不得宣称绝对防逆向，只能说明源码不交付、能力隔离、静态加密和商业级防篡改。

### 8.6 向量化研究与事件驱动回测

- **BT-001** 系统必须提供 Python/DuckDB 向量化研究路径和 Rust 事件驱动回测路径。
- **BT-002** 两条路径必须共享数据标识、费用规则、指标定义和结果 Schema。
- **BT-003** Rust 引擎必须以 MarketEvent 驱动 Strategy、Risk、OMS、Fill Model、Ledger 和 Metrics。
- **BT-004** 同一套 OMS、账本和组合逻辑必须用于历史回测、实时本地模拟和 Demo Trading。
- **BT-005** K 线回测必须避免前视偏差，只有在 K 线结束后才能使用完整 K 线数据。
- **BT-006** K 线内同时触及多个价格条件时必须使用文档化的保守成交顺序。
- **BT-007** 逐笔模式必须按事件时间推进并支持部分成交、费用、滑点和延迟。
- **BT-008** 实时模式可以使用 L2 估算可成交数量和滑点，但不得声称精确模拟队列位置。
- **BT-009** 系统必须支持 Market、Limit、Post-only 和 IOC 四种订单类型。
- **BT-010** 系统不得支持杠杆、做空、借币和交易所高级算法订单。
- **BT-011** 每次 RunSpec 必须固定数据校验和、质量状态、代码哈希、参数、随机种子、初始资产、引擎版本、连接器版本、费用、滑点、延迟和成交模型。
- **BT-012** 回测结果不可原地覆盖；修改输入必须创建新运行。
- **BT-013** 相同 RunSpec 必须产生相同结果哈希，非确定性来源必须被禁止或显式记录。
- **BT-014** 报告必须包含收益、波动率、Sharpe、Sortino、最大回撤、Calmar、胜率、盈亏比、换手率、费用、滑点和敞口。
- **BT-015** 每份报告必须展示数据质量、模拟精度、假设和未运行检查。

### 8.7 本地模拟、OMS、组合与风险

- **SIM-001** 用户必须能够创建独立的本地模拟账户。
- **SIM-002** 模拟账户必须保存现金、资产、持仓成本、未完成订单、成交和权益曲线。
- **SIM-003** 默认报告币种为 USDT，用户可以选择 USDC。
- **SIM-004** OMS 必须使用客户端幂等订单 ID，并实现明确的订单状态机。
- **SIM-005** OMS 必须处理拒单、部分成交、完全成交、撤单和未知结果。
- **SIM-006** 风控必须支持最大订单金额、最大单币仓位、最大总敞口和每日亏损限制。
- **SIM-007** 过期行情、余额不足、异常价格或无效交易规则必须拒绝 OrderIntent。
- **SIM-008** 系统必须提供全局 Kill Switch 和 Cancel All。
- **SIM-009** Local Simulation 与 Exchange Demo 必须在界面和日志中使用明确不同的环境标识。
- **SIM-010** OKX 和 Bitget Demo 必须同步账户、订单、成交和余额，并执行启动及定期对账。
- **SIM-011** Demo 下单出现网络超时且结果未知时，系统必须查询订单状态，不能直接重发。
- **SIM-012** Demo API Key 必须存入 OS Keychain，并与生产环境配置类型隔离。

### 8.8 AI 研究助手

- **AI-001** AI Gateway 必须支持 OpenAI-compatible 和 Anthropic-compatible 两种协议。
- **AI-002** OpenAI-compatible 必须优先支持 Responses API，并兼容 Chat Completions。
- **AI-003** Anthropic-compatible 必须支持 Messages API。
- **AI-004** 用户必须能够配置 Base URL、模型名称和 API Key。
- **AI-005** HTTP Base URL 只允许 localhost 或 127.0.0.1；远程 URL 必须使用 HTTPS。
- **AI-006** 系统必须支持流式输出、取消、连接测试和模型选择。
- **AI-007** API Key 必须存入 OS Keychain，不能进入 React、SQLite、日志或提示词。
- **AI-008** AI 默认无权读取本地文件、数据库和回测结果。
- **AI-009** 用户必须按会话授权具体代码、数据摘要或回测结果，并能随时撤销。
- **AI-010** 发送前必须展示上下文范围，并扫描密钥、令牌和 CEX 凭据。
- **AI-011** AI 只能解释、总结和生成代码草稿，不能执行回测、修改正式策略包或创建订单。
- **AI-012** 默认日志只能保存提供者、模型、耗时、Token 数量和错误，不保存完整提示词或回答。
- **AI-013** AI 服务故障不得影响市场数据、回测、模拟交易或应用启动。

### 8.9 任务、日志与恢复

- **TASK-001** 下载、导入、回测、策略测试和对账必须实现持久化任务状态机。
- **TASK-002** 标准状态必须包含 Queued、Running、Completed、Failed、Cancelled、Interrupted 和 Resumable。
- **TASK-003** 应用退出时不得把未完成任务标记为成功。
- **TASK-004** 可恢复任务必须保存检查点；不可恢复任务必须保留输入和诊断信息。
- **TASK-005** 用户必须能够暂停、取消、恢复任务，并查看阶段进度。
- **TASK-006** Worker 崩溃必须被检测并隔离，关联任务必须进入 Interrupted 或 Failed。
- **TASK-007** 磁盘空间不足时必须停止写入并通知用户，不得自动删除已有数据。
- **TASK-008** 日志必须结构化、分级并统一脱敏。
- **TASK-009** 登录、授权、数据修改、策略运行、回测和订单操作必须进入追加式审计记录。

### 8.10 用户界面

- **UI-001** V1.0 必须提供 Overview、Markets、Data Center、Research、Factors、Strategies、Backtests、Simulation、Portfolio & Orders、AI Assistant、Tasks & Logs 和 Settings 模块。
- **UI-002** 数据完整性、数据来源、连接器状态和模拟精度必须在相关页面可见。
- **UI-003** 系统不得静默切换交易所、时间周期、模型、数据源或成交模型。
- **UI-004** Local Simulation 与 Exchange Demo 必须具有明显且一致的视觉区分。
- **UI-005** 长任务必须在后台执行，界面保持可操作并展示进度。
- **UI-006** 错误信息必须包含用户可执行的恢复动作，不能只展示原始堆栈。
- **UI-007** 应用应当支持简体中文和英文，并以稳定的消息键实现国际化。

## 10. Implementation Decisions

### 10.1 技术栈

- Desktop：Tauri 2。
- Frontend：React、TypeScript strict、Vite、shadcn/ui、Tailwind CSS。
- Router/Server State/Table：TanStack Router、Query、Table。
- Local Core：Rust，异步运行时和并行任务由 Rust 生态实现。
- Research：Python SDK 和 DuckDB，不在正式策略运行路径中嵌入 Python。
- Operational Store：SQLite。
- Historical Store：Parquet。
- Analytical Query：DuckDB。
- Component Runtime：Wasmtime 和 WebAssembly Component Model。
- Authentication：Supabase Auth、Postgres RLS 和设备会话。
- Secrets：macOS Keychain、Windows Credential Manager 等系统密钥库。

V1.0 不引入 Go、Redis、RabbitMQ、TimescaleDB 或 Kubernetes。

### 10.2 接口原则

- 前端与 Rust 使用类型化 Tauri Command/Event，不使用本地公开 HTTP API。
- 外部远程服务采用版本化 REST/WebSocket 协议。
- 交易所适配器通过统一 Port 暴露产品、历史、实时和 Demo 能力。
- 统一模型保留 Venue/NativeSymbol 和必要原生字段，不制造虚假的跨交易所一致性。
- 数据 Schema、RunSpec、Research Manifest、WIT 和策略包均独立版本化。
- 未来真实交易、中心数据湖和 Trader Store 必须通过新 Adapter 接入，不得绕过现有领域边界。

### 10.3 任务模型

- 所有长任务均具有唯一 ID、输入哈希、状态、进度、检查点和诊断信息。
- 任务队列使用本地持久化和有界内存通道，不使用外部消息队列。
- 任务输出必须先写临时文件、校验完成后再原子发布到正式目录。

### 10.4 数据迁移

- SQLite 和 Parquet Schema 必须显式版本化。
- 应用升级前必须验证可用磁盘空间并备份关键元数据。
- 不支持降级读取的新 Schema 必须在启动时阻止运行并给出恢复说明。

## 11. Non-Functional Requirements

### 11.1 正确性与确定性

- 账本、费用、持仓和订单状态转换必须使用精确数值类型。
- 回测不能依赖墙上时钟、无种子随机数或未记录的外部请求。
- 数据质量降级必须传播到研究和回测报告。
- 任何“成功”状态必须意味着输出文件已校验且元数据已提交。

### 11.2 性能

- WebView 不得执行历史数据扫描、回测或 WASM 计算。
- 实时引擎必须消费完整事件流，UI 可以按固定刷新频率降采样展示。
- 内部队列必须有界，并暴露积压、丢弃和处理延迟指标。
- 数据下载、Parquet 写入、DuckDB 查询和回测必须建立固定参考数据集基准。
- 发布前必须证明在项目定义的参考硬件上，前台交互在后台同步和回测期间保持可用。
- 具体吞吐阈值应由 M2 建立的可重复基准锁定，之后性能回退作为 CI 发布门槛。

### 11.3 可靠性

- REST 仅对限流、超时和临时服务端错误执行带抖动的指数退避。
- Demo 下单必须使用幂等 ID，并对未知结果执行查询确认。
- WebSocket 重连后必须重新订阅并重建需要状态的流。
- 磁盘满、权限不足和损坏文件必须失败关闭，不能继续产生部分有效结果。
- 关键元数据写入必须具有事务性，数据文件发布必须具有原子性。

### 11.4 安全与隐私

- Tauri Capability 按窗口最小授权，WebView 不加载不受信任远程页面。
- 应用必须采用严格 CSP 和结构化输入验证。
- 应用更新、策略包和发布资产必须验证签名。
- 日志、崩溃报告和遥测不得包含密钥、Token、完整提示词和用户策略内容。
- 崩溃报告默认仅保存在本地；上传必须获得用户确认。
- 安全承诺不覆盖已经完全控制操作系统和用户账户的攻击者。

### 11.5 可维护性

- 模块通过稳定、小型接口隔离，业务规则不得散落在 React 组件和连接器中。
- OKX 和 Bitget 必须通过同一连接器契约套件。
- 交易所响应 Fixture 和黄金回测数据必须可离线运行。
- 公共 Schema 和 SDK 必须遵循兼容性政策并提供迁移说明。

### 11.6 可访问性与国际化

- 核心功能必须可通过键盘操作。
- 状态不能只依赖颜色表达。
- 文案使用消息键并支持简体中文和英文。
- 时间统一存储为 UTC，展示时允许用户选择时区。

## 12. Testing Decisions

### 12.1 测试原则

- 测试外部可观察行为、领域不变量和接口契约，不绑定内部实现细节。
- 固定数据、固定时钟、固定种子和离线 Fixture 是普通 CI 的默认方式。
- 依赖真实 OKX/Bitget Demo 的测试单独运行，外部服务波动不能阻塞每次提交。
- 所有回归测试失败必须能定位到数据、规则、引擎或连接器版本。

### 12.2 必测模块

- 认证、设备注册、撤销、RLS 和 7 天离线宽限。
- Decimal、费率、账本、持仓和组合不变量。
- 产品规则映射、限流、签名和连接器错误分类。
- L2 快照/增量重建、缺口、乱序和重连。
- 数据下载、去重、质量标记、Parquet Schema 和 SQLite 目录。
- Python Manifest 与 Rust/桌面结果模型兼容性。
- K 线保守成交、逐笔回放、部分成交、滑点和延迟。
- OMS 状态机、幂等 ID、未知结果恢复和对账。
- WIT 兼容性、包签名、恶意 Component、Fuel、内存和超时限制。
- AI 协议适配、URL 安全、上下文授权和密钥脱敏。
- 任务取消、检查点、Worker 崩溃和应用重启恢复。
- 安装、升级、Schema 迁移、卸载和用户数据保留行为。

### 12.3 发布检查

- 前端严格类型检查和生产构建。
- Rust 格式、Clippy 和 Workspace 测试。
- Python lint、类型检查和测试。
- SDK/WIT 兼容套件。
- 依赖漏洞、许可证、SBOM 和密钥泄露扫描。
- macOS 与 Windows 安装、升级和回滚验证。

## 13. Error Handling Decisions

- 错误必须具有稳定代码、用户消息、技术原因、可重试性和建议动作。
- 工具错误不得被解释为市场数据或交易事实。
- 无法修复的数据缺口必须保留证据并降低数据集质量等级。
- AI、单个交易所或单个 Worker 的故障必须局部隔离。
- 系统不得为了“继续运行”而静默降低数据精度或禁用风控。
- 删除、迁移和覆盖操作必须先展示影响并要求确认。

## 15. 风险与缓解

### 15.1 CEX API 频繁变化

缓解：版本化连接器、契约测试、原生字段保留、官方变更日志监控和 Fixture 回归。

### 15.2 本地全市场数据规模和限流

缓解：关注列表优先、渐进后台同步、分区 Parquet、断点续传、全局速率预算和明确覆盖率。

### 15.3 回测结果虚假精确

缓解：分层成交模型、保守假设、质量与精度徽标、不可变 RunSpec 和禁止历史队列位置承诺。

### 15.4 Python 环境碎片化

缓解：版本化 SDK、锁定模板、标准 Manifest、外部环境自检和明确兼容矩阵。

### 15.5 WASM 生态和 WASI 版本变化

缓解：以自定义 WIT 为稳定契约，固定受支持 Wasmtime 版本，不把 WASI 0.3 作为策略接口本身。

### 15.6 本地 IP 保护有限

缓解：源码不交付、签名、静态加密、设备绑定、内存内解密、能力限制和诚实安全声明；更强机密计算留待后续版本。

### 15.7 单人开发范围风险

缓解：纵向切片、OKX 优先、月度发布门槛、Bitget 后接入、范围冻结和禁止提前开发真实交易及商城。

### 15.8 地区和 API 可用性

缓解：展示交易所地区限制和 API 错误，不绕过地理或账户限制；用户负责遵守交易所条款和所在地区规则。

## 17. Further Notes

- “支持多 CEX”在 V1.0 中指 OKX 与 Bitget 的标准化接入和契约兼容，不表示跨交易所智能路由或流动性聚合。
- “模拟交易”包括本地确定性模拟和官方 Demo Trading，不包括真实资金交易。
- “策略保护”表示源码不交付和商业级防篡改，不表示可以对本机完全控制者提供绝对机密性。
- “全市场数据”表示两家支持交易所的活跃现货交易对，并受官方 API 可用性、地区限制和限流约束。
- V1.0 架构必须为未来中心数据湖、真实交易服务和 Trader Store 保留接口，但不得提前实现这些范围外能力。
